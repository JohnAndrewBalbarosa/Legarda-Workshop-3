# EC2 Workshop Automation Platform

## Overview
This project is a local-first, React-based automation tool designed to guide workshop participants through step-by-step EC2 configuration on the AWS Console Home Page. It leverages Playwright (Chromium) and the VS Code browser agent to overlay interactive UI elements (e.g., blinking highlights, navigation cues) directly on the AWS Console Home Page, ensuring users know exactly where to interact. The system synchronizes all participants with a presenter, allowing real-time pacing and progress tracking, and provides an usher interface for targeted assistance.



## Key Features
- **Automated, Visual AWS Console Guidance:**
  - Playwright-driven overlays (blinking/highlighted UI, step-by-step navigation) on the AWS Console Home Page.
  - React frontend for user interaction and step display.
- **Presenter Synchronization:**
  - Presenter controls workshop pacing (next/previous step) via a local web interface.
  - All participant browsers listen for presenter signals (webhook over LAN) and update steps in real time.
- **Progress Reporting:**
  - Presenter dashboard shows which users have completed each step.
  - Exportable reports for post-event analysis.
- **Usher Interface:**
  - Usher dashboard is not a local deployment; it connects to the presenter server and only receives concerns from users.
  - Usher dashboard displays participant status (current step) and seat/table numbers, and receives information about which step each participant is stuck on, along with their seat/table number, so they can provide targeted assistance.
  - Usher can also send a "concern resolved" signal to the presenter for post-report analysis.
- **Assistance Requests:**
  - Participants can request help via a button, notifying ushers with seat/table info.

## Technical Architecture
- **Frontend:**
  - React (Vite or Create React App)
  - Playwright (Chromium) for browser automation and UI overlays
- **Backend/Sync:**
  - Node.js/Express server (runs locally on presenter and each participant)
  - WebSocket (ws) for real-time communication (presenter → participants, participants → presenter)
  - Each participant's app attempts to auto-connect to the presenter's rarely-used private IP (e.g., 10.250.250.1) and port. If unable to connect, a QR code fallback is available. The presenter can generate this QR code on demand (via a button) if the default IP is unavailable or occupied.
  - Each deployment model (user, presenter, usher) is deployed and managed independently, and they communicate strictly via networking (API/WebSocket), never via direct function calls or shared code.
- **Networking:**
  - All devices connect over local WiFi (no internet required)
  - Presenter hosts a WebSocket server on a static, rarely-used private IP (e.g., 10.250.250.1) and port (e.g., 5050)
  - Usher interface can run on a separate port/IP or as a separate route
- **Data Storage:**
  - In-memory or lightweight local file (JSON/SQLite) for progress tracking and reporting

## Implementation Suggestions
### 1. Playwright & VS Code Browser Agent Integration
- Use Playwright in headful mode with Chromium to control and overlay the AWS Console Home Page.
- Leverage the VS Code browser agent to navigate the AWS Console Home Page, retrieve links, and determine which elements to highlight or blink.
- The agent observes both its own navigation and the developer's manual navigation, taking notes on which sites and elements are clicked for documentation and deterministic automation.
- Inject React components as overlays using Playwright's page.evaluate and expose functions for step navigation.
- Use Playwright's selectors to highlight and blink specific AWS Console Home Page elements.

### 2. Real-Time Sync (WebSocket)
- Use the ws library for WebSocket communication.
- Presenter runs a WebSocket server; participants connect as clients.
- On LAN, use a static IP (e.g., 10.250.250.1) for the presenter. This is in the "Class A" private range and is highly unlikely to be assigned by DHCP.
- On startup, participant apps attempt to connect to ws://10.250.250.1:5050. If unable to connect, participants can use a QR code fallback generated by the presenter (only if the presenter chooses to generate it via a button). QR/manual entry is only used if the default IP is unavailable.
- For usher, run a separate WebSocket server or use a different port (e.g., 5051) for the usher dashboard.

### 3. Progress Tracking & Reporting
- Each participant sends step completion events to the presenter server.
- Presenter dashboard aggregates and displays progress in real time.
- Store progress in a local file (JSON or SQLite) for later export/analysis.

### 4. Assistance Requests
- Participants can click a "Need Help" button, sending a message to the presenter/usher dashboard with their seat/table number.
- Usher dashboard displays outstanding help requests and allows filtering by seat/table.

### 5. Usher Interface
- Usher dashboard connects to the presenter server (or a dedicated usher server/port).
- Displays participant status (current step) and outstanding help requests, along with seat/table numbers.
- Usher receives information about which step each participant is stuck on, so they can go directly to the participant needing assistance.

### 6. Security & Local Networking
- No authentication required; all communication is local.
- Use CORS and firewall rules to restrict access to the local network.
- Optionally, allow the presenter to delegate another IP/port for usher or backup presenter.

## Getting Started
1. Clone the repository to each device (presenter, participants, ushers).
2. Run per-role setup scripts:
   - Presenter machine: `cd presenter; npm run setup`
   - User machine: `cd user; npm run setup`
   - Optional Playwright browser install for user: `cd user; npm run setup:playwright`
3. Start presenter host (WebSocket/API + presenter UI):
   - `cd presenter; npm run start`
   - Presenter UI URL: `http://PRESENTER_IP:5050/presenter`
4. Start user host on participant device:
   - PowerShell: `cd user; $env:PRESENTER_WS='ws://PRESENTER_IP:5050'; npm run start`
   - User UI URL: `http://127.0.0.1:5174/user?id=user-1&seat=A1&ws=ws://PRESENTER_IP:5050`
5. Start user AWS guide (auto-opens AWS target URLs and blinking selectors):
   - PowerShell: `cd user; $env:PRESENTER_WS='ws://PRESENTER_IP:5050'; npm run start:aws-guide`
6. Use usher as link-only dashboard (no Playwright required):
   - Open from any device/browser: `http://PRESENTER_IP:5050/usher?id=usher-1&ws=ws://PRESENTER_IP:5050`
   - If usher device cannot reach presenter WS directly, use the same URL but replace `PRESENTER_IP` with reachable LAN IP.
7. For developer self-host with Playwright (headful Chromium):
   - Presenter: `cd presenter; npm run start:playwright`
   - User panel: `cd user; $env:PRESENTER_WS='ws://PRESENTER_IP:5050'; npm run start:playwright`
8. All devices must be on the same WiFi network.
